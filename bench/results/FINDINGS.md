# Wirehair optimization — findings

Target: AMD Zen 5 (Threadripper PRO 9985WX, AVX-512/GFNI/VPOPCNTDQ, 128 HW threads).
Harness: `bench/whx.cpp` (modes: micro, bench, fuzz, repro, ohead, scan, seedsearch),
built via `bench/build.sh` with ablation flags. Ablations tracked in beads (`bd list`).

## 1. SIMD GF(256) — kernels much faster, but ~0 end-to-end (codec is MEMORY-BOUND)

### S1 — GFNI `gf2p8affineqb` constant-multiply (poly 0x14D)  [beads wirehair-cqu]
- Replaced split-nibble `pshufb` mul/muladd with one `vgf2p8affineqb` per 64 B (precomputed
  256× 8×8 affine matrices). **Bug found & fixed during bug-hunt**: Intel SDM uses `M.byte[7-i]`,
  so output bit j → matrix byte (7-j); true GFNI identity is `0x0102040810204080` (a widely-cited
  value, `0x8040201008040201`, was wrong). Exhaustive self-check (256 consts × 256 vals × 7
  offsets) + 89k end-to-end fuzz round-trips: 0 failures.
- **Kernel speedup (clean, 64 threads, L1/L2-resident):** `mul_mem` 2.0–2.8×, `muladd_mem` 1.5–2.2×.

### S2 — AVX-512 ZMM XOR widening of add/addset/add2_mem  [beads wirehair-1sa]
- 64-B ZMM, 4× unrolled (add2/addset were previously un-unrolled single-YMM loops).
- **Kernel speedup (clean, 64 threads):** add/addset/add2 ~1.3–1.76× at 4–16 KB; ~1.0× ≥64 KB
  (memory-bandwidth-bound). Self-check + 287k fuzz round-trips: 0 failures.

### The decisive result: SIMD doesn't move end-to-end speed
- `perf stat`, single-thread, bb=8192 N=4096, baseline vs S1+S2:
  - instructions **5.90B → 3.35B (−43%)**, but cycles **6.28B → 6.16B (−1.8%)**, IPC **0.94 → 0.54**,
    L1-miss ~22%. Same null result at 1 thread and at 128 threads.
- Profiling (bb=1300 N=2048 and N=16000): **~56–58% of self-time is data-block XOR**
  (`gf256_add_mem`/`add2_mem`/`addset_mem`), ~13% peeling (pointer-chasing). `gf256_muladd_mem`
  (GFNI target) <2%; GE bit-matrix ops (`MultiplyDenseRows`/`Triangle`, G1 target) <1%.
- **Conclusion:** Wirehair is memory-bound (scattered access over a multi-MB recovery-block
  working set), not GF-compute-bound. Faster GF kernels reduce instructions/power but not wall time.
  G1 (SIMD GE bit-matrix XOR) closed as not-worth-it for the same reason. [beads wirehair-p0t]
- **Disposition:** S1/S2 are correct and gated behind `-DWH_GFNI=1` / `-DWH_AVX512=1` (off by
  default). They help GF-compute-bound regimes (cache-resident data, many shared-nothing streams)
  and cut instruction count, but are not a default end-to-end win on this codec.

### Measurement methodology notes (important)
- Benchmark MUST reuse codec objects (`wirehair_*_create` reuseOpt) + buffers + warmup, or
  allocation/page-faulting dominates (~25–37% of time) and masks codec compute.
- Multi-thread end-to-end numbers are noisy (memory-bandwidth contention); use interleaved
  single-thread median-of-N for clean A/B.

## 2. Overhead — a real, systematic seed-table defect (fix in progress)  [beads wirehair-ai7]

- Wirehair's mean overhead is ~N+0.02 as documented, BUT a scan of real-world overhead
  (random-erasure, `ohead`/`scan` modes) found a **cluster of weak-spot N where overhead is
  5–150× normal** — e.g. N=3061 mean **0.50** (vs 0.02), N=3430 **3.0** (max 744 extra blocks),
  N=2962 **2.1**. Concentrated in N≈2000–10000 (the `kPeelSeeds[N%2048]` + interpolated-dense regime).
- **Root cause:** specific bad `(peel_seed, dense_count, dense_seed)` combinations for individual N.
  Each N's peel seed only affects that N, and **almost any non-default peel seed fixes it**
  (e.g. N=3061: 0.50 → 0.015). The siblings sharing the same `kPeelSeeds` index are fine, so it's a
  per-N table-quality issue (insufficient seed-search at table-generation time).
- **Fix (bug-safe):** per-N peel-seed correction table `kPeelSeedFixups[]` (binary-searched at the
  top of `GetPeelSeed`, included from `WirehairPeelFixups.inc`). Generated by a parallel
  `seedsearch` campaign that, per weak N, searches candidate seeds and verifies the replacement on
  independent loss patterns at loss=0.10 and 0.30. Validated end-to-end (weak N fixed, good N
  unchanged → no overfitting; correctness fuzz preserved). Full enumeration + table generation in
  progress.

## Note on the mid-run reboot
The box hard-reset once under load: a marginal DIMM (slot **P0MH0**, recurring fatal DRAM
"map-out" CPER history) tripped under an over-subscribed run (two 96–102-thread campaigns at once,
~200 threads on 128 HW threads). Not OOM, not a software panic. Mitigations applied: hard thread
cap (`hw-16`, ≤104) in the harness; one saturating campaign at a time; small blocks (low DRAM
bandwidth) for the overhead campaigns. Recommended: memtest/replace P0MH0; install
`rasdaemon`/`ipmitool` for live error capture.

## 3. Overhead seed-defect fix — SHIPPED & VERIFIED  [beads wirehair-ai7]

Scanned real-world overhead (random erasures) across N and found a systematic seed-table
defect: **581 weak-spot N in [2048,24000]** where overhead is 5–150× normal (301 with mean
>1.0 vs documented 0.02). Nature: bimodal — most decodes fine, but ~0.1–0.5% of erasure
patterns are *catastrophic* (up to 816 extra blocks). Each is a peel-seed defect (any non-default
seed fixes it; siblings sharing the kPeelSeeds index are fine). Built a **228-entry per-N peel-seed
correction table** (`WirehairPeelFixups.inc` → `GetPeelSeed`, binary-searched, default-on, bug-safe).

Verified [2048,17500]: weak (mean>0.10) **306→80**, severe (>0.3) **221→60**, **0 regressions**,
correctness fuzz **0 fails / 66,624 trials** (mean overhead back to 0.022). Residual ~80 need joint
peel+dense seed search (catastrophic at high loss) — Task5. Range [17500,64000] — Task4.

## 4. Task 6 (the real speed lever) — MEASURED & CONCLUDED

Added `WH_COUNT` per-stage symbol-XOR instrumentation. `encoder_create` moves **13.0× the message
size in XOR** (GF-multiply is 0.4%). Breakdown: **Substitute 5.5× + PeelDiagonal 3.7× +
MultiplyDenseValues 2.5×** dominate; the Gaussian elimination is only **1.3×** — so codex's
GE-focused ideas (Markowitz/inactivation) have limited ceiling. The 13× is **memory-latency-bound**
(IPC 0.5–0.9, L1-miss 13%, LLC-miss 13%, dTLB-miss 8.7%).

Memory micro-optimizations tried — **all null** (gated off, documented):
- **Software prefetch** (cross-row in Substitute): ~0% — Zen5 OoO already overlaps the misses.
- **Huge pages** (madvise MADV_HUGEPAGE): ~0% — bound by data misses, not TLB.
- **LTO**: ~0% — hot loops already single-TU.

**Conclusion:** Wirehair is near its memory-bound structural limit. The only real speed lever is
**reducing the 13× amplification** — a lighter degree distribution / dense-count (fewer symbol XORs,
trades decode overhead) or a different code structure — which requires full offline table
regeneration (beads explorations epic). No free speed wins remain at the implementation level.

## 5. Speed — DEFINITIVELY CLOSED (proven 3 ways)

The peel-weight-cap experiment quantified the degree-distribution lever: cap=30 gives ~8% fewer
XORs and a MEASURED ~5% faster create+decode (N=4096), for +0.005 overhead. BUT a feasibility
test (seedsearch under cap=30) showed the lighter peeling **fundamentally breaks invertibility**:
large N (16000, 32000) have NO working peel seed (encoder fails for every seed); N=7301/10000 stay
unsolvable. The lighter peeling overwhelms the fixed dense+heavy precode → to fix it you must GROW
the precode (more dense rows), which adds back the saved traffic. The degree distribution is deeply
coupled to the precode; the 13× amplification is a near-optimal balance.

**Speed is closed:** (1) SIMD null end-to-end (memory-bound), (2) memory micro-opts null (HW
already handles the access), (3) the only structural lever (lighter distribution) can't reduce net
traffic without breaking the code. A faster Wirehair would require a fundamentally different
fountain-code construction — out of scope for an optimization pass. The shipped value is the
overhead fix; SIMD paths remain available (gated) for cache-resident regimes.

## 6. Large-N overhead defects are WORSE (Task4, in progress)

Extending the scan past 17500 found the seed defects are MORE severe at large N: in [24000,24500]
alone, N=24405 has mean overhead **59** (max 8861 extra blocks!), N=24146 mean 12.3. seedsearch
fixes most (24405: 14.8→0.015, 24091: 0.20→0.020); a minority need dense-seed co-tuning (Task5).
Extension campaign over [17500,32000] running; the per-N correction-table method scales.

## 7. Task4 partial: overhead fix extended to [2373,24384]

## 8. Task5: dense-seed co-tuning (joint peel+dense search)

Built a 2-stage joint search: peel-search first; for peel-UNfixable N (where no peel seed works,
esp. at loss 0.30), search dense seeds too. Validated: N=24146 5.4->0.021 (peel=35,dense=38).
Ran on 294 hard N (cut at 153, [4935,25211]). Tables: peel 601 entries + NEW dense 59 entries
(WirehairDenseFixups.inc -> GetDenseSeed). Build clean, correctness fuzz 0 fails. Catastrophic
defects fixed via dense seed: N=7533 17.7->0.018, N=11446 8.5->0.016 (@loss 0.30). Known gap:
GOOD30=0.15 accepts mediocre peel fixes over better dense fixes for some N (e.g. 24146->0.133);
tighten goodthr+re-run for full quality. Remaining hard N [25211,31912] use the same method.

## 9. FINAL overhead-fix result (headline)

Tables: 574 peel + 62 dense corrections ([2373,25787]), default-on, bug-safe.

  ORIGINAL: mean overhead 0.0893, max 2907 extra blocks
